Deep Dives

What To Learn In Each Pillar

Concrete skills, concepts, and technologies

Pillar 01

LLM Fundamentals

Foundation

Before you can build on top of models, you need a working mental model of what they are and how they behave. You don't need a PhD — but you need enough to reason about failures, costs, and design trade-offs.

"A software engineer who treats an LLM as a black box will be outcompeted by one who understands its nature, tendencies, and failure modes."

Key concepts to internalize: how autoregressive generation works, what a context window is and why it matters, the difference between training vs inference, what temperature and sampling parameters do, why models hallucinate, and how model size relates to capability and cost.

  • Transformer architecture basics
  • Tokens, embeddings, and vocab
  • Context windows & attention
  • Hallucination causes & patterns
  • Temperature, top-p, top-k
  • Training vs inference costs
  • Multimodal inputs (vision, audio)
  • Model benchmarks & evaluation
Pillar 02

Prompt Engineering & Context Design

Critical

Prompt engineering is the most immediately valuable skill. It is, in essence, programming in natural language — and it requires the same rigor as writing clean code. System prompts define agent behavior. Few-shot examples replace test suites. Structured outputs replace function signatures.

Beyond basics, learn advanced techniques like chain-of-thought (CoT), tree-of-thought (ToT), ReAct patterns, self-consistency, and role-playing prompts. Learn how to write prompts that are robust to adversarial inputs and edge cases.

  • System prompt design
  • Chain-of-thought prompting
  • Few-shot & zero-shot strategies
  • Structured output (JSON, XML)
  • ReAct (Reason + Act) pattern
  • Self-consistency sampling
  • Prompt versioning & testing
  • Context window management
  • Output parsing & validation
  • Meta-prompting techniques
Pillar 03

Agentic Systems Design

Critical

This is the frontier. Agentic systems are AI applications that take sequences of actions, use tools, make decisions, and accomplish multi-step goals with minimal human intervention. They are more complex than standard software because they are non-deterministic, stateful, and self-directing.

Learn the core loop: perceive → plan → act → observe → repeat. Understand memory types (in-context, external, episodic), tool calling patterns, and how to structure agents that fail gracefully and can be interrupted or corrected.

  • Agent loop architecture
  • Tool/function calling
  • Memory systems (short/long term)
  • Multi-agent orchestration
  • Planning & task decomposition
  • Subagent delegation patterns
  • MCP (Model Context Protocol)
  • State management in agents
  • Error handling & retries
  • Human-in-the-loop design
"The hardest part of agentic design isn't making agents powerful — it's making them fail safely and predictably."
Pillar 05

Data & Retrieval (RAG)

Critical

Retrieval-Augmented Generation (RAG) is the dominant pattern for giving agents access to private, large, or up-to-date knowledge bases. It's become a core engineering discipline unto itself, with substantial complexity around chunking, embedding quality, retrieval strategy, and re-ranking.

Master the full pipeline: document ingestion → chunking → embedding → vector storage → retrieval → augmentation → generation. Then learn advanced patterns like HyDE (Hypothetical Document Embeddings), multi-hop retrieval, and graph-based RAG.

  • Embedding models & spaces
  • Vector databases (Pinecone, Weaviate, pgvector)
  • Chunking strategies
  • Semantic vs keyword search
  • Hybrid search (BM25 + dense)
  • Re-ranking (Cohere, cross-encoders)
  • Document parsing pipelines
  • Metadata filtering
  • Graph RAG
  • Eval frameworks for RAG
Pillar 06

AI Safety & Evaluation

Critical

Every engineer shipping AI systems is now responsible for a new category of risks that didn't exist in traditional software: hallucinations, prompt injection, biased outputs, harmful content, and autonomous actions with real-world consequences. Evals are the new unit tests.

Learn to design comprehensive eval suites, measure output quality, catch regressions, and implement guardrails. Understand prompt injection attacks — where malicious content hijacks agent behavior — and how to defend against them architecturally.

  • Evals design & frameworks
  • LLM-as-judge patterns
  • Hallucination detection
  • Prompt injection defense
  • Content safety filters
  • Output guardrails (NeMo, etc.)
  • Red-teaming LLM systems
  • A/B testing for LLMs
  • RLHF & constitutional AI concepts
  • Fairness & bias measurement
Pillar 07

MLOps & AI Infrastructure

High

LLM APIs make it easy to get started, but production AI systems require infrastructure thinking: latency budgets, cost per inference, rate limiting, caching strategies, model routing, and observability. At scale, a 10ms latency improvement or 5% cost reduction can matter enormously.

Also understand fine-tuning: when to fine-tune vs. prompt engineer, the mechanics of LoRA and QLoRA, and how to manage training data pipelines. These are increasingly within reach of application engineers, not just ML researchers.

  • LLM observability (Langfuse, etc.)
  • Latency & cost optimization
  • Prompt caching
  • Model routing & fallbacks
  • Fine-tuning (LoRA, QLoRA)
  • Serving frameworks (vLLM, TGI)
  • Token budgets & batching
  • Streaming responses
  • Model distillation basics
  • CI/CD for AI pipelines

Mental Model

The Mindset Shift Required

How to think differently as an AI-era engineer

The hardest part of transitioning isn't learning new APIs — it's unlearning mental models that served you well in traditional software engineering but actively hinder you when building with AI.

Traditional Software Mindset

  • Deterministic: same input = same output
  • Code defines behavior completely
  • Unit tests cover correctness
  • Errors are exceptions to handle
  • Performance = speed & memory
  • Debug by stepping through code
  • Features are added through code
  • One "right" way to implement

AI-Era Engineering Mindset

  • Probabilistic: outputs vary per run
  • Behavior emerges from instructions + data
  • Evals measure quality distributions
  • Failures are expected, design for graceful degradation
  • Performance = quality + latency + cost
  • Debug by analyzing traces & logs holistically
  • Features added through prompt + data changes
  • Explore multiple approaches, measure outcomes
"In traditional software, you write logic. In AI engineering, you write intent — and then verify that the system understood and acted on that intent correctly."

Three new principles to internalize: Measure everything (evals before shipping, always), Assume failure (agents will make mistakes; design for it), and Iterate fast (the prompt/eval loop is now your dev cycle).

What Doesn't Change

Core SE skills that matter more than ever

A common mistake is assuming the AI era makes foundational software engineering obsolete. The opposite is true — strong software fundamentals become a competitive advantage because many "AI engineers" lack them.

📐
System Design

Designing scalable, resilient distributed systems remains essential. AI just adds new components — models, queues, vector DBs — to the architecture.

🔬
Debugging & Problem Solving

Tracing down subtle bugs in non-deterministic AI systems actually requires better debugging skills than traditional software.

🗣️
Communication

Explaining AI system behavior, limitations, and risks to stakeholders is a critical and undervalued skill.

🧪
Testing Mindset

Evals are the evolution of unit tests. Engineers who think carefully about correctness and coverage win.

📚
Data Structures & Algorithms

Still relevant for performance-sensitive components and for reasoning about search, indexing, and retrieval systems.

🔐
Security Thinking

AI introduces new attack surfaces (prompt injection, data leakage). Security-first thinking is more important, not less.

The best time to start was yesterday.
The second best is today.

The AI Agents Era isn't a future state — it's happening now. Engineers who build fluency in these skills today will define what software looks like for the next decade. Don't try to learn everything at once. Pick a pillar, ship a project, build an eval, repeat.
